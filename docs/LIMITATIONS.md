# Known Limitations

Read this before demoing or evaluating the project â€” it's a direct, unhedged list.

## This pass had no live infrastructure access either
This build session (the Faculty Management / multi-class-upload / email-verification-hardening update) had
no Supabase project, no SMTP provider, no Anthropic/embedding API keys, and no GitHub push access â€” same
constraints as the original build. Note that this repository *has* since been connected to a real Supabase
project and deployed (see `campus-learn-seven-modifications.md` for that investigation), so live infrastructure
does exist for this app now â€” this session specifically just didn't have credentials for it. That means, for
everything added in this pass:
- **The four new migrations (`0019`â€“`0022`) have never been applied to a real Postgres instance.** They were
  written carefully, cross-checked against the exact existing schema (table/column names, existing RLS
  helper functions), and `scripts/gen-types.mjs` was extended to parse them (see `CHANGELOG.md`) as an
  indirect structural sanity check, but `supabase db push` was never run. Apply them to a staging project
  first â€” see `MIGRATION_NOTES.md`.
- **The multi-file/multi-class upload flow, Faculty Management invite flow, and email verification cooldown
  have never round-tripped through real Supabase Auth/Storage/Postgres.** The code compiles, typechecks, and
  the pure authorization logic (`batch-rules.ts`, `verification-cooldown.ts`) is unit tested, but the signed
  Storage upload URL flow in particular (`createSignedUploadUrl` / `uploadToSignedUrl`) has only been verified
  against the Supabase JS SDK's TypeScript types, not exercised against a live bucket.
- **AI features remain code-complete but unexercised** (unchanged from the original build â€” see below).
- **`database.types.ts` is generated by `scripts/gen-types.mjs`**, not the Supabase CLI against a live
  project. This pass found and fixed two real gaps in that generator (missing `IF NOT EXISTS` tables, no
  `ALTER TABLE ADD COLUMN` support) that had been silently producing incomplete types since an earlier
  session â€” regenerate with `supabase gen types typescript` against your real project once linked.

## Feature-level limitations
- **PPTX text extraction is not implemented** (`src/lib/ai/document-extraction.ts` throws a clear error for
  `.pptx` MIME types). PPTX files can still be uploaded, downloaded, and viewed as resources â€” they're just not
  processed for AI summarization/RAG. PDF, DOCX, and TXT extraction are implemented.
- **Staff and Faculty invitations create the account directly with a generated temporary password**
  (`inviteStaffAction`, and the new `inviteFacultyAction` added in this pass), rather than emailing an invite
  link, since no SMTP/transactional email provider is configured in this environment. Unlike the original
  `inviteStaffAction`, `inviteFacultyAction` does show the temporary password once, in the Administrator's
  browser only, with a clear "will not be shown again" warning â€” but it still requires the Administrator to
  relay it to the Faculty member through some secure channel outside the app. Configure Supabase custom SMTP
  (or switch to `admin.auth.admin.inviteUserByEmail`, which requires SMTP too) to get a true passwordless
  email-invite flow.
- **The multi-file upload flow has no scheduled cleanup job for orphaned staged objects.** If a browser tab
  crashes or loses network *after* some files finish uploading to Storage but *before* `finalizeUploadBatchAction`
  is called, those objects are never referenced by any `resources` row (so no student ever sees them) but also
  are not automatically deleted â€” they sit under `resources-batch/{subjectCode}/{batchId}/...` until manually
  cleaned up. A production deployment should add a scheduled job (Supabase Edge Function cron, or a Vercel
  cron route) that removes `resource_upload_batches` rows (and their Storage objects) left in `status =
  'uploading'` for longer than, say, 24 hours.
- **Upload progress is per-file, not per-byte.** The multi-file upload UI shows each file as
  pending/uploading/uploaded/failed, not a byte-level progress bar, because `uploadToSignedUrl` in the pinned
  `@supabase/storage-js` version does not expose upload progress events; a byte-level bar would require
  hand-rolling the multipart PUT request against the signed URL, which was judged too risky to get right
  without a live environment to test it in.
- **Rate limiting falls back to in-memory/per-process only if `UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN`
  are unset** (`src/lib/rate-limit/index.ts` now supports Upstash Redis directly â€” set those two env vars in
  your Vercel project for durable, multi-instance-safe rate limiting; this includes the email verification
  resend cooldown's IP-based limiter, though the cooldown itself is durable in Postgres regardless via the
  `email_verification_requests` table added in this pass).
- **Full-text/global search across all content types** (resources, subjects, announcements, previous-year papers,
  discussions in one query with highlighting) is implemented per-module (resource list filtering,
  discussion list) rather than as a single unified `/search` route with cross-entity ranking. The database has the
  necessary indexes (`resources_search_idx`, `discussions_search_idx`) for this to be built on top of.
- **Notification delivery is in-app only** â€” the `notification_preferences.email_enabled` field and per-category
  toggles exist in the schema and Settings UI, but no email-sending integration is wired up to actually honor them.
- **Descriptive/short-answer quiz questions are stored and displayed but not auto-scored** (by design â€” they
  require Faculty manual review, per spec); the manual-review grading UI for descriptive answers specifically is not
  built (assignment grading is; quiz descriptive-answer grading is the same underlying `quiz_answers.marks_awarded`
  column but lacks a dedicated Faculty review screen).
- **Duplicate question detection** for the Faculty question bank is not implemented.
- **CSV export of quiz results** is not implemented.

## What would change if this were a longer engagement
Broader modules (discussions, announcements, notifications) are implemented with real CRUD and real RLS but less UI
polish than the resource-library/quiz/assignment core. A follow-up pass would add: bulk operations, richer search
with facets, email notifications, a proper staff-invite-by-email flow, PPTX extraction, and a distributed rate
limiter â€” none of which change the architecture, since the seams (validation schemas, permission matrix, RLS) are
already in place for them.

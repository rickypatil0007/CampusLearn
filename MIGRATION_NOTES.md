# Migration Notes

## What's new

Four new, purely additive migrations: `0019_faculty_teaching_assignments.sql`,
`0020_multi_file_multi_class_uploads.sql`, `0021_email_verification_requests.sql`,
`0022_rls_audit_and_scoping.sql`. Nothing in `0001`–`0018` was modified. They are safe to run against a
Supabase project that already has `0001`–`0018` applied (the state described in
`campus-learn-seven-modifications.md`), and safe to run from scratch on a brand-new project as part of the
full `0001`–`0022` sequence.

## Applying against a live Supabase project

```bash
npx supabase login
npx supabase link --project-ref <your-project-ref>
npx supabase db push
```

This applies any migration not yet recorded as run, in order. It will only apply `0019`–`0022` if
`0001`–`0018` are already applied; otherwise it applies the full sequence.

## Regenerating types afterward

```bash
npx supabase gen types typescript --project-id <your-project-ref> > src/types/database.types.ts
```

Do this against the real project once it's linked — the committed `database.types.ts` in this zip was
generated by the repository's own `scripts/gen-types.mjs` (which parses the migration SQL directly, since
this session never had live Supabase credentials to run the command above against). `gen-types.mjs` was
itself fixed in this pass (see `CHANGELOG.md`) to actually pick up `IF NOT EXISTS` tables and
`ALTER TABLE ... ADD COLUMN` statements, so if you keep using it instead of the Supabase CLI, it is now
materially more accurate than before — but the Supabase CLI command above is still the source of truth once
you have a live project.

## Manual Supabase dashboard steps this pass does NOT automate

- **Custom SMTP**: `Authentication → Emails → SMTP Settings`. Supabase's built-in email sending has low
  rate limits and is not suitable for production verification/reset email volume. Faculty invitations in
  this pass create the account directly with a generated temporary password rather than a true email
  invite, specifically because this environment has no SMTP credentials to send one — see
  `docs/LIMITATIONS.md`.
- **Auth redirect URLs**: `Authentication → URL Configuration` must include your deployed origin
  (`https://your-domain/**`), or email verification/reset links will redirect incorrectly.
- **Verification link expiry**: `Authentication → Emails → Email OTP Expiration` (or the equivalent field
  in your Supabase dashboard version) — set to `86400` seconds (24 hours) per spec section 6.3. This is a
  project-level Auth setting; it is not something a migration can set.
- **Storage bucket confirmation**: migration `0020` raises the `resources` bucket's `file_size_limit` to
  ~95MB and expands `allowed_mime_types`. Confirm this took effect in
  `Storage → resources → Configuration` after running `db push` — bucket settings occasionally need a
  dashboard refresh to reflect a SQL-level update.
- **Optional: Upstash Redis** for distributed rate limiting (`UPSTASH_REDIS_REST_URL` /
  `UPSTASH_REDIS_REST_TOKEN` in your Vercel project's env vars). Without it, `src/lib/rate-limit/index.ts`
  falls back to an in-memory limiter that does not hold across concurrent serverless instances.

## Rollback

Every new migration in this pass is additive (`create table if not exists`, `add column if not exists`,
`create policy`/`drop policy if exists` pairs). To roll back cleanly, drop the four new tables
(`faculty_teaching_assignments`, `resource_upload_batches`, `resource_classes`,
`email_verification_requests`) and re-run the `0010`/`0018` policy definitions for `resources` and
`announcements` to restore their pre-`0022` (broader, unscoped) `SELECT`/`INSERT` policies. No data in any
pre-existing table is deleted or mutated by these migrations, only new columns and rows.
